# vcpkg в проектах CMake

## Постановка проблемы

При использовании жёсткого указания путей, проект может не быть переносимым на другие Windows-машины, на которых библиотка Boost будет расположена в другой папке. Пример настройки создающих проблемы сборки на других машинах:

``` CMakeList.txt
if (WIN32)
	# Boost размещается не стандартным образом, в не стандартных местах расположения на жёстком диске и
	# функция find_package найти его не сможет. По этой причине, CMake нуждается в подсказке
	set( BOOST_ROOT "C:/Program Files/vcpkg/installed/x64-windows" )
	set( BOOST_INCLUDEDIR "C:/Program Files/vcpkg/installed/x64-windows/include/boost" )
	set( BOOST_LIBRARYDIR "C:/Program Files/vcpkg/installed/x64-windows/lib" )    
endif (WIN32)
```
**Вариант решения:** vcpkg предлагает простую интеграцию с CMake, чтобы сделать установленные пакеты доступными в проектах автоматически.

Механизм интеграции vcpkg заключается в предоставлении файла цепочки инструментов CMake.

**Наши шаги поэтапно:**

1. Создаём файл конфигурации `CMakePresets.json` (создаем его в одной директории с CMakeList.txt), который будет ссылаться на файл цепочки `toolchain` . Внутренности:

```json
{
    "version": 3,
    "configurePresets": [
        {
            "name": "default",
            "binaryDir": "${sourceDir}/build",
            "cacheVariables": {
                "CMAKE_TOOLCHAIN_FILE": "$env{VCPKG_ROOT}/scripts/buildsystems/vcpkg.cmake"
            }
        }
    ]
}
```

где `"$env{VCPKG_ROOT}/scripts/buildsystems/vcpkg.cmake"` указание пути расположения `vcpkg.cmake`. Данная часть является изменяемой в зависимости от расположения.

Соответственно в блоке кода представленного выше (if (WIN32)....endif (WIN32)) необходимости нет, его можно убрать.

> Файл конфигурации должен находиться рядом с CMakeList.txt

2. Настраиваем переменную окружения VCPKG_ROOT (один раз на компьютере). 
   
```shell
set "VCPKG_ROOT=C:\Program Files\vcpkg"
   или без кавычек
set VCPKG_ROOT=C:\Program Files\vcpkg
```

 Это нужно для того, что бы не вводит каждый раз полный путь.
   
```shell
cmake -B build -S . -DCMAKE_TOOLCHAIN_FILE="C:/Program Files/vcpkg/scripts/buildsystems/vcpkg.cmake"
```
   
> Обратите внимание что переменная `VCPKG_ROOT` содержится в файле `CMakePresets.json`

 **!** Для сохранения переменной навсегда добавляем её в переменное окружение Windows.
   
   
4. Важный шаг, который забывать не надо это группирование проекта (конфигурация) 

   ```shell
   cmake --preset default
   ```

   Данная команда запускает конфигурацию CMake с использованием предустановленных настроек из файла CMakePresets.json. Её необходимо использовать до основного `cmake ..`

   > Когда вы установите переменную окружения в переменное окружение Window всё равно есть вероятность ошибки выполнения команды конфигурации, в таком случае рекомендуется в окне терминала повторно настроить  переменную окружения VCPKG_ROOT
   
5. Далее идёт стандартный набор команд
   
```shell
mkdir build
cd build
cmake ..
cmake --build .
```

ВАЖНЫЕ ИЗМЕНЕНИЯ:
У Boost вышло много обновлений и последнее добавленное мной было ( и есть на данный момент) boost 1.89. 
Первоначально Boost планировалось скачивать с официального сайта Boost.org, но из-за временой блокировки было приянто решение сделать это через менеджер пакетов vcpkg.
Вскоре выяснилось что в vcpkg кривая копия boost (а именно необходимый в другом проекте boost::beast), и т.к официальный сайт boost.org был дотсупен скачивание прошло через него. После прехода на Boost 1.86 возникли следующие проблемы:
1. Проблема сборки в CMake: В СMake указывался путь к boost/.../.lib, что при использовании версии 1.86 является неккоректным, т.к в версии 1.86 теперь используются header-only
    **Наш случай:**
   В Boost 1.74: Компонент system был исключительно скомпилированной библиотекой.Ему физически требовался файл boost_system.lib на Windows или libboost_system.so на Linux. Без указания COMPONENTS system в CMake проект не линковался.
   В Boost 1.89: Разработчики полностью переписали system. Теперь это header-only (чисто заголовочная) библиотека.Физического файла .lib для неё больше не существует.
   Когда CMake по старой привычке искал бинарник system, он закономерно выдавал ошибку, так как искать было нечего.
2. CMake-конфигурация
В Boost 1.74: Поиск происходил «по старинке» через встроенный в CMake текстовый модуль FindBoost.cmake. Он просто сканировал папки на жестком диске.
В Boost 1.89: Новые версии Boost поставляются с собственными конфигурационными файлами (например, BoostConfig.cmake), которые пытаются перехватить управление. В сочетании с менеджером пакетов vcpkg (который у нас установлен) это приводило к тому, что CMake игнорировал наши локальные файлы и пытался стянуть новейший Boost из интернета или кэша vcpkg.
3. Зависимость program_options и date_time от компиляции
   В Boost 1.74: Все три компонента (system, date_time, program_options) требовали компиляции (создание .lib)
   В Boost 1.89: date_time тоже стал практически полностью заголовочным. А вот program_options всё ещё требует компиляции(файла .lib)(мы используем разбор командной строки).
Основная проблема при составлении грамотного CMakeLists.txt на мой взгляд состоит в параллельном использовании Boost и vcpkg(по прежнему используется для библиотеки spdlog).
